Working Group meeting 24/03/2023

Date: 24/03/2023
Participants:, Natalie Muric, Peter Borresen
Model editor: Andreea PasÄƒre
Note editor: Jana Ahmad

Agenda

  • Discuss the eFulfilment model with respect to the CEN-TC-440-Peppol-Despatch advice model

    • To continue from Consignee information.

Discussion

The eFulfilment model was discussed and compared with the CEN-TC-440-Peppol-Despatch advice model in order to identify any differences or areas for improvement. Several changes were made to the eFulfilment model to align it with the Peppol standard, as follows.

All parties and roles are covered in eFulfilment model * Shipment information: is covered in eFulfilment model

  • For Consignment information in Peppol requirements:

    • Handling code: a listcode is created

    • Handling Instruction and identifier are covered in eFulfilment model

  • The "dct:Description" property has been added to "epo-ful:ShipmentAgreement" to cover shipment description requirements.

  • Good Item: Declared statistics value amount

  • The property TransportHandlingUnit is deleted from dispatch identifier class

  • Carrier in Peppol requirement

    • Carrier party is covered i eFulfilment model in the level of consignee, epo-ful:ShipmentStage uses epo-ful:TransportMeans which has a driver person epo-ful:TransportMeansOperator.

  • Driver identification document type:

    • Epo:hasDocumenttype property is added

  • eop-ful:hasOperatorLicienceId relation is added

    • For Transport mode code, Transport means type code requirements.

  • transport mode is covered by epo-ful:transport-mode

  • At-voc-new:transport-means codelist is added

  • epo-hasOperatorLicenceID association is added.

  • For Train identifier requirement:

    • transport means id is updated with identifier;

  • Delivery information and Delivery term location: are covered in eFulfilment model

  • epo-ord:specifiesPlaceOfStorage is deleted.

  • epo-ord:concernsPlaceOfDelivery is changed to epo-ord:specifiesPlaceOfDelivery

  • Delivery tracking identifier and period:

    • In ePo, there are hasTrackingId, deliveryPeriod properties

  • Delivery location

    • In ePo, there is dct:Location.

  • Estimated delivery period requirement:

    • epo-ful:hasEstimatedDeliveryPeriod in epo-ful:Shipmentinformation level

  • Actual or expected despatch period requirement:

    • epo-ful:dispatchDate

  • Despatch address requirement

    • epo-ord:specifiesPlaceOfStorage is removed and replaced with epo-ful:specifiesPlacesofDespatch.

    • During the discussion, it was pointed out that the concept of a virtual location may be necessary in cases where the order pertains to a service rather than a physical product.
